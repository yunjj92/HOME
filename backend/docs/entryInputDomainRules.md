# 수입/지출 데이터 입력 도메인 규칙

## 목적

예산관리에 필요한 로우데이터 입력 기능의 도메인 규칙을 정의한다.
입력 대상은 공통 거래 테이블 `entries`와 거래 유형별 상세 테이블 `incomes`, `expenses`이다.

## 거래 구분

- 거래는 수입과 지출로 구분한다.
- 프론트엔드 요청에서는 `entryType`으로 거래 구분을 전달한다.
- `entryType`이 수입이면 `entries`와 `incomes`에 저장한다.
- `entryType`이 지출이면 `entries`와 `expenses`에 저장한다.

## 금액

- `amount`는 항상 0 이상의 정수로 입력한다.
- 지출 금액도 음수로 저장하지 않는다.
- 수입/지출 여부는 금액 부호가 아니라 `entryType`으로 판단한다.
- 0원 거래는 허용한다.
- 소수점 금액은 허용하지 않는다.
- 통화는 거래가 연결된 계좌의 `currencyType`을 따른다.

## 날짜

- `date`는 실제 거래일을 의미한다.
- 화면에서는 일자까지만 입력한다.
- 미래일 입력을 허용한다.
- 저장 시 Asia/Seoul 기준 00:00:00으로 변환하여 저장한다.

## 계좌

- 모든 거래는 `account_id`를 필수로 가진다.
- 엑셀 업로드 시 사용자가 선택한 계좌를 업로드 데이터 전체에 일괄 적용한다.
- 삭제된 계좌는 선택 목록에 노출하지 않는다.

## 수입

- 수입은 `source_id`를 필수로 가진다.
- 미분류 수입은 허용하지 않는다.
- `source_id`는 입금자명 또는 수입원 데이터를 의미한다.
- 새로운 입금자명이 나오면 `sources`에 등록한 뒤 `incomes.source_id`에 저장한다.
- 예산 편성은 별도 화면에서 처리한다.
- 추후 예산 편성 화면에서는 수입으로 잡힌 금액 중 아직 편성되지 않은 금액을 계산할 수 있어야 한다.

## 지출

- 지출은 `merchant`를 필수로 가진다.
- 지출은 `ministry_id`를 필수로 가진다.
- `tag_id`는 선택값이다.
- `fixed_expenses` 연결은 입력 단계에서 처리하지 않는다.
- `contributions` 연결은 입력 단계에서 처리하지 않는다.

## 메모

- `entries.memo`는 선택값이다.
- 비고 성격의 값으로 사용한다.

## 중복 거래

- 중복 거래는 기본적으로 저장을 막는다.
- 구체적인 중복 판단 기준은 추후 세부 설계에서 정의한다.

## 수정 및 삭제

- 입력 후 거래 수정이 가능하다.
- 입력 후 거래 삭제가 가능하다.
- 삭제는 soft delete로 처리한다.
- `entries` 삭제 시 연결된 `incomes` 또는 `expenses`도 함께 soft delete한다.
- 거래 수정/삭제에는 별도 제한을 두지 않는다.

## 엑셀 업로드

- 은행에서 반출되는 양식을 기준으로 처리한다.
- 은행별 parser를 따로 구현한다.
- 엑셀 업로드 시 계좌는 사용자가 선택한 계좌를 적용한다.
- 업로드 원본 데이터는 일단 별도 보관하지 않는다.

## 자동 분류

- `thesauruses` 테이블을 자동 분류 사전으로 사용한다.
- `merchant`를 기준으로 `ministry_id`, `tag_id` 등을 자동 추론한다.
- 자동 추론 결과는 사용자가 수정할 수 있다.
- 자동 분류는 보조 기능이며 필수값은 아니다.

## 저장 단위

- 입력대기목록 전체를 한 번에 저장한다.
- 하나라도 실패하면 전체 저장을 롤백한다.
- 저장 전 프론트엔드와 백엔드 양쪽에서 validation을 수행한다.
